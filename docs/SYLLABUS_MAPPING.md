## Part A — API Ecosystem & FinTech Integration

| # | Syllabus topic | Where it lives | Notes |
|---|----------------|----------------|-------|
| A1 | API ecosystem & lifecycle | `services/payments/app.py`, `nginx/nginx.conf` | Versionable REST surface fronted by an API gateway; OpenAPI auto-generated by FastAPI at `/api/pay/docs`. |
| A2 | Microservices architecture | `services/payments`, `services/neural`, `services/rag` + `docker-compose.yml` | Three independently deployable services with their own Dockerfile, port and data store. |
| A3 | FinTech integration · payments | `services/payments/ledger.py` | Double-entry ledger in SQLite (WAL). Every operation produces balanced legs; trial balance must net to zero on every `/health` call. |
| A4 | ISO 20022-style message identifiers | `ledger.py` `transfer()` | Each P2P transfer returns an `end_to_end_id` (`e2e_<ts><rand>`), mirroring the `EndToEndId` field of an ISO 20022 `pacs.008` payment instruction. |
| A5 | Sync vs async | `app.py` (sync REST) + `frontend/_loopcards.js.html` sync-along loop | Recharge / transfer / redeem are synchronous and idempotent; the player's listening sync runs on a 60-second async timer that redeems 1 minute at a time. |
| A6 | Event-driven design | `app.py` → `archive_to_s3()` | Every payment txn emits a JSON event to the S3 raw zone; downstream Glue jobs consume the event stream (event sourcing pattern). |
| A7 | Idempotency | `Idempotency-Key` header on POST routes; `idempotency` SQLite table | Replaying the same key returns the original response — required for safe retries in payments. |
| A8 | API security · JWT | `services/payments/security.py` | Dependency-free HS256 JWT with iat/exp claims; secret from `LOOPY_JWT_SECRET` env. |
| A9 | API security · OWASP & headers | `nginx/nginx.conf` `add_header` block | `X-Content-Type-Options nosniff`, `X-Frame-Options SAMEORIGIN`, `Referrer-Policy`, `Permissions-Policy`, `server_tokens off`. |
| A10 | OAuth-style bearer flow | `POST /auth/token` + `Authorization: Bearer …` middleware | Simple OAuth2 password-grant shape; pluggable to Cognito/Auth0. |
| A11 | mTLS readiness | `nginx/nginx.conf` (commented) + Terraform SG | The gateway is the TLS termination point; the SG allows only the gateway port externally so service-to-service traffic stays on the internal docker network. |
| A12 | API gateway · routing | `nginx/nginx.conf` `location /api/*` blocks | Path-based routing with prefix stripping (`/api/pay/foo` → upstream `/foo`). |
| A13 | API gateway · throttling | `limit_req_zone api` + `limit_req_zone auth` | Token-bucket rate limit: 20 r/s general, 5 r/s on `/auth/token`, return 429 on overflow. Live-tested in `docs/RUNBOOK.md`. |
| A14 | Performance · compression | `nginx/nginx.conf` `gzip on` block | gzip for JSON / JS / CSS / SVG above 512 B. |
| A15 | Performance · caching headers | `nginx/nginx.conf` static block | Static assets served with `try_files` from the gateway pod; Cache-Control extendable via `expires` directive. |
| A16 | Performance · rate limiting | A13 | Same mechanism, different lens — protects upstream services from runaway clients. |
| A17 | Protobuf / binary serialisation | *Out of scope, JSON chosen for demo*; documented in `docs/ARCHITECTURE.md` § "What we deliberately didn't build". |
| A18 | Scalability · load balancing | `infra/terraform/main.tf` `aws_lb` + `aws_lb_target_group` + `nginx/nginx.conf` `upstream` blocks | **Real AWS Application Load Balancer** across two AZs distributing across N app EC2s (default 2). Each EC2 also runs its own internal nginx load balancer — two layers of LB. |
| A19 | Scalability · fault tolerance | `docker-compose.yml` `restart: unless-stopped` + ALB target-group `/healthz` probe + `services/payments/app.py` `/health` trial-balance probe | Crashed containers restart; ALB stops routing to unhealthy targets within ~30 s; CloudWatch alarm fires on `UnHealthyHostCount > 0`. |
| A20 | Observability · structured logs | `services/payments/security.py` `JsonFormatter` + nginx `log_format json` | All app and gateway logs are single-line JSON — directly ingestible by CloudWatch Logs Insights. |
| A21 | Observability · metrics | `services/payments/app.py` `/metrics` (Prometheus text format) | Per-process counters: `loopy_pay_txn_total`, `loopy_pay_errors_total`, plus request count. |
| A22 | GraphQL | *Out of scope, REST chosen.* Mentioned in `docs/ARCHITECTURE.md` as a future extension; the ledger's normalised reads (`/cards`, `/journal`) are GraphQL-ready. |

## Loop Cards — the bespoke FinTech feature

The "Loop Cards" feature is the project's headline contribution and ties
together many of the syllabus points above:

* **Graphically fabulous** — six hand-tuned gradient-mesh themes (aurora,
  midnight, sunset, matrix, candy, gold) with a moving sheen, chip and grain
  (`frontend/_loopcards.css.html`).
* **Shareable** — `/cards/{id}/share` adds a friend to `shared_with`; both the
  owner and shared friends can redeem from the card.
* **Recharge in 60-minute blocks** — `MINUTES_PER_RECHARGE = 60`; costs
  `COINS_PER_RECHARGE = 120` Loopy Coins.
* **Sync along** — the frontend's playback loop calls `/redeem` once a minute
  so the timer on the card matches the music actually being listened to.
* **Auditable** — every action is two balancing ledger entries; the
  `/cards/{id}/journal` endpoint shows the full trail.
